# SMAC EDU 랜딩페이지 제작 가이드

특정 페이지가 아니라 **앞으로 만들 모든 특강 랜딩페이지에 공통으로 적용**해야 하는 패턴 모음. 사진특강 랜딩페이지(`photo-lecture-landing.html`)를 만들면서 겪은 문제와 해법을 일반화한 것.

---

## 1. 신청 폼 필드 규칙

모든 신청 모달은 아래를 기본값으로 한다:

- 이름, 이메일: 필수
- **연락처: 필수**, `010-1234-5678` 형식 검증 (정규식 `^01[0-9]-\d{3,4}-\d{4}$`)
- **개인정보 수집·이용 동의 체크박스: 필수** (미체크 시 제출 불가)
- 하단 안내문: "*제출 버튼을 클릭하시면 개인정보 수집 및 이용에 동의한 것으로 간주됩니다."
- 동의 체크박스의 실제 응답은 백엔드에서 가짜로 "동의함"을 채우지 말고, 사용자가 실제로 체크한 값만 그대로 기록한다.

## 2. 관리자 알림은 매번 보내지 말고 배치로

신청자 본인에게 가는 확인 메일은 매번 즉시 발송하되, **운영자에게 가는 알림 메일은 누적 신청자 수가 10의 배수가 될 때만(10명, 20명, 30명...) 발송**한다. 매번 알림이 가면 신청이 몰릴 때 정신없다는 피드백을 받았음. 호출될 때는 단순 통보가 아니라 최근 10명을 표로 정리해서 보여준다.

## 3. 전환율을 위한 "신청만 해도 선물" 패턴

- 랜딩페이지의 히어로 CTA 버튼과 하단 CTA 버튼 아래, 둘 다에 "🎁 신청만 해도 [무언가] 드립니다" 문구를 반복 배치 (핵심장점 반복 전략)
- 신청 확인 메일에 실제 선물(PDF 자료 등) 다운로드 버튼과, 있다면 카카오톡 단체채팅방 입장 버튼을 추가
- 선물·채팅방 링크는 강의마다 다르므로 매번 사용자에게 확인

## 4. Open Graph 미리보기 이미지는 처음부터 만들어둔다

OG 메타태그가 없으면 카카오톡 등 메신저가 페이지 안의 임의 이미지(보통 강사 사진처럼 가장 눈에 띄는 img 태그)를 미리보기로 가져가 버린다. 매번 다음을 한다:

1. 후킹 카피 + 일정 + 선착순 + 선물 문구를 담은 1200×630 배너를 헤드리스 Chrome 스크린샷으로 직접 제작한다 (디자인 HTML을 만들고 `chrome --headless --window-size=1200,630 --screenshot=...`로 캡처).
2. `<head>`에 다음을 추가:
   ```html
   <meta property="og:type" content="website" />
   <meta property="og:url" content="https://www.smacedu.kr/[페이지명]" />
   <meta property="og:title" content="..." />
   <meta property="og:description" content="..." />
   <meta property="og:image" content="https://www.smacedu.kr/images/[배너].png" />
   <meta property="og:image:width" content="1200" />
   <meta property="og:image:height" content="630" />
   <meta name="twitter:card" content="summary_large_image" />
   ```
3. **카카오 캐시 주의**: 카카오는 링크 미리보기를 한동안 캐싱한다. 배포 후 바로 확인하려면 URL 끝에 `?v=2` 같은 더미 쿼리스트링을 붙여서 공유한다 (페이지는 동일하게 서빙되고, 카카오만 "새 URL"로 인식해서 새로 긁어감). 정식 URL은 며칠~몇 주 후 자연스럽게 캐시가 풀리며 갱신된다.

## 5. 모바일 브라우저의 날짜/전화번호 자동 감지를 반드시 의심한다

**증상**: 페이지에 적어둔 날짜·시간 문구(예: "2026년 7월 1일(수) 저녁 9시")가 모바일에서만 숫자 부분이 이상하게 보임 — 삼성 인터넷·카카오톡 인앱 브라우저에서는 숫자마다 칩 박스가 둘러쳐지고, 다른 브라우저에서는 숫자만 흐리게/다른 색으로 표시됨. 데스크톱에서는 정상으로 보여서 놓치기 쉽다.

**원인**: 전화번호를 자동으로 통화 링크로 바꿔주는 것과 같은, 모바일 브라우저의 "스마트 텍스트 감지" 기능이 날짜 패턴을 인식해서 자기 스타일로 다시 그려버리는 것. 페이지 CSS/디자인 문제가 아니라 방문자 브라우저가 콘텐츠를 가로채는 문제다.

**효과 없었던 방법** (순서대로 시도했지만 실패):
1. `<meta name="format-detection" content="telephone=no, date=no, address=no, email=no">` — iOS Safari 표준 방법이지만 삼성 인터넷은 따르지 않음.
2. 숫자와 단위(년/월/일/시) 사이에 zero-width space(`&#8203;`) 삽입해서 정규식 패턴을 깨뜨리는 시도 — 시각적으로는 동일하지만 삼성 인터넷에는 효과 없었음.

**확실하게 통하는 방법**: 날짜/시간이 강조되어야 하는 배지·헤드라인 같은 짧은 텍스트는 **처음부터 PNG 이미지로 만들어서 `<img>`로 넣는다.** 이미지 안의 텍스트는 브라우저가 "글자"로 인식할 수 없어서 자동 감지 자체가 원천적으로 불가능해진다.

- 이미지는 헤드리스 Chrome으로 디자인 HTML을 스크린샷해서 만들면 됨 (실제 CSS 폰트·색상 그대로 재현 가능)
- `<img>`에는 반드시 `alt`로 같은 텍스트를 넣어 접근성·SEO 보존
- `style="width: 100%; max-width: [원래 배지 너비에 맞는 값]px; height: auto; display: block;"`로 반응형 처리
- 본문 문단처럼 줄바꿈이 자유로워야 하는 긴 텍스트에는 이 방법을 쓰지 말고, 짧고 고정된 배지·헤드라인류에만 적용

**테스트 시 주의**: 로컬 헤드리스 Chrome에 `--window-size=390,900` 같은 좁은 값을 줘도 `viewport meta`의 `width=device-width`가 항상 기대대로 적용되지 않아서, 모바일이 아니라 데스크톱 레이아웃을 그냥 잘라서 보여주는 경우가 있었다. 모바일 렌더링 확인은 **실제 기기 스크린샷이 훨씬 신뢰할 수 있는 방법**이다.

## 6. Apps Script로 신청 처리할 때

- 신청 데이터를 모을 스프레드시트가 이미 정해져 있다면(공식 신청서 등), **그 시트에 직접 바인딩된 전용 스크립트를 새로 만드는 편이 기존 공용 스크립트를 찾아 고치는 것보다 빠르고 안전**하다. 이 저장소에는 Apps Script가 여러 개의 독립 프로젝트로 흩어져 있어서, 어떤 게 실제 운영 중인 스크립트인지 식별하는 데 시간이 걸렸음.
- 코드를 수정해도 git push만으로는 적용되지 않는다 — Apps Script 편집기에서 직접 코드 갱신 후 **배포 → 배포 관리 → 새 버전 → 배포**가 별도로 필요하다는 것을 매번 사용자에게 안내한다.

## 7. 작업 검증 방법

- HTML 변경 후에는 태그 짝(div/section/script/style) 개수가 맞는지 빠르게 확인하고, 헤드리스 Chrome으로 스크린샷을 찍어 섹션별로 실제 렌더링을 눈으로 확인한다.
- `.reveal` 스크롤 애니메이션 때문에 단순 스크린샷은 요소가 흐리게/안 보이게 찍힐 수 있다 — `--virtual-time-budget`을 충분히 주거나, 테스트용 사본에 `.reveal{opacity:1!important}`를 임시로 덧붙여서 확인 후 삭제한다.
- 신청 폼 동작은 실제로 한 번 제출해보고, 시트에 행이 쌓이는지 + 확인 메일이 오는지 + (10명 단위라면) 관리자 알림이 오는지까지 확인한다.
